# CLAUDE.md — RF4 트렌드 프로젝트 가이드

이 문서는 모든 작업(매니저·구현·검수)이 공유하는 기준이다. 세션 시작 시 자동으로 읽힌다.

## 프로젝트 한 줄 요약

러시안피싱4(RF4) 공식 주간기록(rf4game.kr)을 15분마다 수집·누적해, 등록한 선호 어종의 **활성도**(지금 잘 잡히는지)를 접속 즉시 추천하는 개인 대시보드. 1인 개발·운영(박지원).

## 아키텍처

- **단일 프로세스**: `app.py` 하나가 FastAPI 웹서버 + 15분 수집 스레드 + 하루 1회 정리를 다 돌린다.
- **스택**: Python 3.12, FastAPI, Jinja2, SQLite(WAL), BeautifulSoup, bcrypt, itsdangerous
- **배포**: 안드로이드 태블릿(Termux 상시 구동) + Cloudflare Named Tunnel → rf4trends.com
- **모듈 경계** (rf4site/):
  - `app.py` — 라우트·서버·백그라운드 스레드. DB 경로는 `RF4_DB` 환경변수(미설정 시 `BASE_DIR/rf4.db`)
  - `collector.py` — 주간기록 파싱·누적 (30개 게시판, 403/429 백오프)
  - `scoring.py` — 활성도 계산 (**교체 가능한 추천 모듈** — 나중에 ML로 통째 교체 예정)
  - `labels.py` — 라벨 수집 + 활성도 스냅샷 박제 (ML 학습 데이터)
  - `auth.py` — 회원 인증
  - `maintenance.py` — 7일 초과분을 archive.db의 bait_records로 분리
  - `templates/` — base/dashboard/species/login/onboarding

## 핵심 설계 원칙 (반드시 지킬 것)

1. **시간은 UTC 단일 기준.** 저장·필터·계산은 전부 UTC, 사람이 보는 표시만 KST 변환. first_seen은 **공백 구분자**(`YYYY-MM-DD HH:MM:SS`)로 저장 — T 구분자(isoformat)로 저장하면 SQLite `datetime('now')`와 문자열 비교가 깨진다(과거 D-37 버그).

2. **수역(낚시터)별 독립 집계.** 게임이 수역별로 독립적으로 돌아가므로, 같은 어종도 수역마다 따로 점수를 낸다. 합산 보정은 자의적 왜곡이라 하지 않는다. 대표값 = 최고 활성도 수역.

3. **"코드버그 vs 기능문제" 구분.** 코드가 잘못된 것(→코드 수정)과, 코드는 맞지만 데이터·설계 특성상 생긴 현상(→코드 안 건드림)을 명확히 나눈다. PRD/CHANGELOG에 태그로 분류.

4. **추천 로직(scoring.py)은 교체 가능하게.** 현재 활성도는 임시 수식. 나중에 LightGBM 트리 기반 순서형 분류로 교체 예정이라, 점수 로직을 한곳에 모으고 다른 곳(특히 JS)에 중복시키지 않는다.

5. **스캐폴딩 패턴 선호.** 권한 토글 등은 값 변경보다 "코드 일부만 지우면 되돌아가는" 구조로 남긴다 (예: `if not (is_admin or not is_admin)`).

## 작업 방식 (이 프로젝트의 일하는 법)

- **결정은 PRD.md에 D-번호로 기록.** 무엇을·왜 + 근거. 현재 D-39까지 있음. 새 결정은 다음 번호로.
- **판단의 서사는 EXPERIENCES.md에 기록.** PRD가 "개별 결정"이라면 EXPERIENCES.md는 "결정들이 어떻게 연결됐나"의 이야기 — 기능을 만들었더니 실제 문제가 드러났고, 과거 기능과 상충해 어떤 근거로 풀었고, 그게 다음으로 어떻게 이어졌는지. 포트폴리오에서 사고력을 보여주는 핵심 자료. 의미 있는 판단/상충이 있을 때만(사소한 건 PRD로 충분).
- **코드 변경 후 반드시 세 스위트 통과 확인.** (tools에서)
- **surgical 변경.** 요청한 것만 건드린다. 인접 코드·주석·포맷을 "개선"하지 않는다. 내 변경으로 생긴 잔재(미사용 import 등)만 정리한다. 기존 죽은 코드는 발견하면 보고만, 멋대로 지우지 않는다.
- **단순함 우선.** 요청 안 한 기능·추상화·유연성·방어코드를 넣지 않는다. senior 엔지니어가 "과하다"고 할 코드면 다시 쓴다.
- **검증 가능한 성공 기준으로 작업.** "검증 추가"가 아니라 "이 입력에 이 결과가 나오는 테스트를 통과시킨다".
- **데이터 출처는 공식(rf4game.kr)만.** 사설 사이트(rf4kr.com, rf4-stat 등)는 코드·문서 어디에도 언급하지 않는다.
- **문서는 한국어로.**

## 함정 메모 (실수하기 쉬운 것)

- **first_seen 포맷**: 반드시 공백 구분자. isoformat(T) 쓰면 시간 필터 깨짐.
- **실데이터 DB로 테스트 시 admin 계정 충돌**: 업로드된 운영 DB엔 이미 'admin'이 있어 signup 400/login 401이 난다. 새 유저명(예: tester99)으로 가입할 것. 유저명이 너무 짧으면(t1 등) validation에 걸린다.
- **tojson 한글**: app.py에서 `templates.env.policies["json.dumps_kwargs"] = {"ensure_ascii": False, ...}` 설정으로 한글이 HTML에 그대로 보이게 함.
- **운영 DB는 함부로 건드리지 않는다**: 계산 방식·표시만 바꾸는 변경은 DB 스키마를 안 건드린다. 마이그레이션이 필요하면 일회성은 코드에 박지 말고 수동 명령으로 안내(단일 운영 환경이므로).

## 에이전트 팀 구성

작업은 역할별 에이전트로 나뉜다. `/feature`로 신기능 워크플로우를 돌린다.

- **매니저(메인)** — 사용자와 직접 대화. 기획·설계, 검수 결과 검토, 배포 결정 오케스트레이션. 위임하지 않고 직접 함.
- **coder** — 확정된 설계를 surgical하게 구현. 자동 위임(구현 단계).
- **reviewer** — 모든 코드 변경 후 자동 검수. 빠짐없이 보고하되 수정·커밋·처리결정은 안 함.
- **frontend-designer** — UI 미적·UX 설계. UI 얽힌 기능에서 매니저가 설계 단계에 호출하거나 사용자가 직접 부름. 공식 `frontend-design` skill을 참고 자료로 활용(설치 시).

문서 작업(PRD 기록, README 패치노트, 이슈 정리)은 **매니저가 직접** 한다 — 별도 작가 에이전트를 두지 않는다(매니저가 방금 한 작업의 맥락을 그대로 갖고 쓰는 게 더 정확).

호출 기준: 코드 구현→coder, 검수→reviewer(자동), 화면 디자인→frontend-designer, 글쓰기·문서→매니저 직접. 작은 변경은 매니저가 바로 처리.

**참고 — skill vs agent**: `frontend-design`은 skill(디자인 지식·가이드), `frontend-designer`는 agent(그 일을 하는 담당자). 충돌 아니라 보완 — 에이전트가 skill을 참고해 일한다. skill 설치: `npx skills add https://github.com/anthropics/skills --skill frontend-design` (Claude Code에서 직접 실행).

## 배포 규칙 (중요)

**어떤 작업도 사용자의 명시적 최종 승인 없이 `git add/commit/push`를 실행하지 않는다.** 검수 통과 → 사용자 검토 → 수정 → 사용자가 "배포" 승인 → 그제서야 커밋. 중간 수정은 작업 트리에만 둔다(사소한 오류까지 커밋 히스토리에 남기지 않는다). 푸시와 태블릿 반영은 사용자 본인이 실행하므로, 명령을 정리해 안내만 한다.
